## 二、核心业务规则

### 用户勾选集合规则

用户通过 UI 勾选一组悬赏等级（如 {A, S}），构成 **activeGrades**，全部业务判定基于集合而非单个级别:

- **招募列表扫描**: 等级图标匹配集合内任意一个 → 判定命中 → 点击加入按钮
- **队伍房间校验**: 房间显示的建议等级匹配集合内任意一个 → 判定通过 → 点击准备
- **完成标记**: 某个等级跑完目标次数后，从 activeGrades 中移除
- **停止条件**: activeGrades 为空 → 所有悬赏已完成 → DONE

### 识别频率

| 模式 | 周期 | 常量 | 适用场景 |
|------|------|------|---------|
| 快速模式 | **100ms** | `FAST_INTERVAL_MS` | 招募列表扫描、战斗检测 |
| 普通模式 | **1000ms** | `NORMAL_INTERVAL_MS` | 导航、等待、结算 |

### 点击频率

| 动作 | 点击后延迟 | 说明 |
|------|-----------|------|
| 加入/准备/导航点击 | **1000ms** (`POST_CLICK_DELAY`) | 等待 UI 过渡动画完成 |
| 关闭弹窗 | 300~800ms | 弹性延迟，视弹窗类型 |
| 返回按钮 | 800~1200ms | 过渡动画较长 |
| 页签刷新 | 200~300ms | 快速切换 |
| 技能释放 | min **500ms** 冷却 | 防止重复触发 |

## 三、完整循环总览

```
runLoop:
  Phase: IDLE/LOBBY/CHAT/RECRUIT_LIST → phaseNavigateAndScan()
    ├─ ensureRecruitView() → 导航到招募列表
    └─ 100ms 全屏检查循环（三块逻辑）
       ├─ ① OUT_OF_RANGE_RECRUIT（超出范围悬赏点击刷新）
       ├─ ② validation（队伍房间校验 + 等级校验 + 准备）
       └─ ③ normal scan（等级图标匹配 + 邀请检测 + TAB刷新）
    return LOBBY / TEAM_ROOM / DONE

  Phase: TEAM_ROOM → phaseValidate()
    └─ 等待 BATTLE_WARNING（15s 超时）
    return FIGHT / SETTLEMENT / LOBBY

  Phase: FIGHT → phaseBattle()
    └─ 检测结算/胜利/失败 + 技能释放
    return SETTLEMENT / LOBBY

  Phase: SETTLEMENT → phaseClaim()
    └─ 关闭结算弹窗 → 领奖 → 回到大厅
    └─ 等级完成 → 从 activeGrades 移除
    return IDLE / DONE

  RECOVERY → delay(1500ms) → IDLE
  异常捕获 → globalFailCount++ → RECOVERY → >=3 → 停止
```

## 四、RECRUIT_LIST 阶段（核心）

### 循环结构

```
100ms loop {

  ── ① OUT_OF_RANGE_RECRUIT 检测 ─────────────
  条件: currentBounty == null && refreshCooldown <= 0
  检测: matchTemplate(OUT_OF_RANGE_RECRUIT, 阈值 0.5)
  命中: → click(检测坐标) → 跳到最新悬赏列表
       → noMatchCycles = 0, refreshCooldown = 10 (1s冷却)
       → continue

  ── ② Validation 校验模式 ────────────────────
  条件: currentBounty != null（已点击加入，进入队伍房间）

  每100ms执行:
    1. DAILY_LIMIT? → exitTeam() → currentBounty = null → continue
    2. SCOPE_RECRUIT 检测（还在招募列表?）
       → 命中 → 加入失败异常 → currentBounty = null
       → return GamePhase.LOBBY（扔回主流程重新导航）
    3. READY_BUTTON?
       └─ 命中 → matchAnyLevelIcon(activeGrades) 对全部勾选等级校验
          ├─ 命中任一 → 点击准备 → return TEAM_ROOM
          └─ 无匹配 → 级别不在勾选范围 → exitTeam() → 重置
    4. 以上皆否 → delay(100ms) → continue

  ── ③ Normal Scan 普通扫描模式 ───────────────
  条件: currentBounty == null

  每100ms:
    1. matchAnyGrade(screen, remaining)
       遍历 remaining（用户勾选且未完成的等级列表）匹配等级图标
       └─ 命中 → currentBounty = grade
               → matchTemplate(JOIN_BUTTON, 全局匹配)
               ├─ 找到 → click → "X悬赏可见，点击加入" → continue
               └─ 未找到 → currentBounty = null → continue

    2. 累计 noMatchCycles++, 达到3周期无匹配:
       a. READY_BUTTON? → 被邀请检测
          ├─ 命中 + 等级匹配 → currentBounty = matchedGrade → continue
          └─ 命中 + 无等级匹配 → exitTeam() → continue
       b. SCOPE_RECRUIT 检测 CHAT_ICON/RECRUIT_TAB?
          → return GamePhase.LOBBY
       c. 仍无变化 → performTabRefresh()
}
```

### 关键判定规则

| 场景 | 判定方式 | 动作 |
|------|---------|------|
| 超出范围悬赏出现 | `OUT_OF_RANGE_RECRUIT` 模板匹配 | 点击该按钮刷新列表 |
| 等级图标匹配 | `matchAnyGrade` 遍历 remaining | 设置 currentBounty → 检测 JOIN_BUTTON |
| 点击加入按钮 | `JOIN_BUTTON` 全局模板匹配 | 点击（可能点到其他悬赏的按钮） |
| 加入后在校验模式 | `currentBounty != null` | 每100ms全量检测 |
| 加入失败（仍在招募列表） | `detectForPhase(SCOPE_RECRUIT)` | 清 currentBounty → 扔回主流程 |
| 已达上限弹窗 | `DAILY_LIMIT` 模板匹配 | exitTeam 退出 |
| 准备按钮出现 | `READY_BUTTON` 模板匹配 | 校验等级后点击 |
| 房间等级校验 | `matchAnyLevelIcon(activeGrades)` | 匹配任一勾选等级 → 准备；都不匹配 → 退出 |
| 被邀请进队 | 3周期无匹配 → `READY_BUTTON` 检测 | 等级校验通过 → 进入校验分支 |
| 3周期无任何匹配 | `noMatchCycles >= 3` | TAB 刷新或重新导航 |

## 五、招募列表刷新

**触发条件:**
1. `OUT_OF_RANGE_RECRUIT` 匹配 → 点击该按钮刷新
2. 连续 3 周期(300ms)无等级匹配 → `performTabRefresh()` 页签切换

**执行序列:**

```
① 检测 CHAT_TAB (templates/chat/private_chat.png)
   匹配 → click → delay(200ms)
   不匹配 → 固定坐标 (0.5f, 0.12f) → delay(200ms)
② 检测 RECRUIT_TAB (templates/chat/team_recruit.png)
   匹配 → click → delay(300ms)
```

**边界:**
- `currentBounty != null` 时不触发 OUT_OF_RANGE 检测
- `refreshCooldown` 提供 1s 冷却防无限循环

---

## 六、等级匹配 → 加入队伍

```
① matchAnyGrade 命中 grade → currentBounty = grade
② 全屏检测 JOIN_BUTTON (minMaxLoc 全局匹配)
   ├─ 命中 → click → "X悬赏可见，点击加入"
   └─ 未命中 → currentBounty = null → 跳过
③ continue → 下周期进入校验模式
```

**规则:**
- 只匹配 `remainingGrades` 内的等级（勾选且未完成）
- `JOIN_BUTTON` 全局匹配，可能匹配到其他悬赏的按钮 → 校验阶段用 `matchAnyLevelIcon` 兜底
- 日志 `"X悬赏可见，点击加入"` 仅表示 X 等级图标可见，**不保证进入了 X 的队伍**

## 七、队伍房间 → 准备

### 进入方式
1. **主动加入**: 匹配等级 → 点击加入 → `currentBounty != null` → 校验分支
2. **被动邀请**: 3周期无匹配 → `READY_BUTTON` + 等级通过 → `currentBounty` 设值

### 准备动作

```
① READY_BUTTON 检测到
② matchAnyLevelIcon(screen, activeGrades)
   ├─ 命中 → log "等级匹配 X (lvX)，点击准备" → click → return TEAM_ROOM
   └─ 无匹配 → "队伍级别不在勾选范围内，退出" → exitTeam() → 重置
```

### 异常处理

| 检测 | 动作 |
|------|------|
| `DAILY_LIMIT` 弹窗 | exitTeam → 重置 currentBounty |
| `SCOPE_RECRUIT` 命中 | 加入失败异常 → return GamePhase.LOBBY 重新导航 |
| 以上皆否 | 保持 100ms 循环等待 |

### 等级校验规则
- 校验用 `matchAnyLevelIcon(screen, ctx.activeGrades)` 而非 `currentBounty`
- 例: 用户勾选 {A, S}，房间显示 lv80 或 lv90 都算匹配
- 不匹配 → 退出 → 回到普通扫描继续找

## 八、TEAM_ROOM 阶段（等待战斗）

```
while (< 15s):
  BATTLE_WARNING? → click → delay(1000ms) → return FIGHT
  WAITING_SCREEN? → continue
  SETTLEMENT_POPUP / CONFIRM_BUTTON? → return SETTLEMENT
  CHAT_ICON / RECRUIT_TAB? → return IDLE
  每 5s → log "等待中..."
15s 超时 → exitTeam() → IDLE
```

- 检测周期: 100ms
- 超时: 15s

---

## 九、FIGHT 阶段（战斗）

```
while (true):
  SETTLEMENT_POPUP → return SETTLEMENT
  CONFIRM_BUTTON → return SETTLEMENT
  DEFEAT_POPUP → return SETTLEMENT
  每 500ms:
    ULTIMATE_SKILL (阈值 0.6) → click
    WEAPON_SKILL (阈值 0.6) → click
  10次(1s) 无匹配 → SCOPE_BATTLE 判定 → IDLE
```

- 检测周期: 100ms | 技能冷却: 500ms

---

## 十、SETTLEMENT 阶段（结算领奖）

```
while (< 300轮):
  SETTLEMENT_POPUP → clickOutside() → delay(800ms)
  CONFIRM_BUTTON → click → delay(1000ms)
  CHAT_ICON → 回到大厅 → break
  3次无匹配 → SCOPE_CLAIM 判定
```

### 完成标记

```
回到大厅后:
  runCounts[grade] += 1
  log "X 完成 N/M"
  if (runCounts >= targetRuns):
    activeGrades = activeGrades - grade    ← 从集合删除
  currentBounty = null
  if (activeGrades.isEmpty()):
    return DONE
  return IDLE
```

**关键规则:**
- 完成后从 activeGrades 移除，后续不再扫描该等级
- 全部移除完 → `activeGrades.isEmpty()` → DONE
- 不需要 OpenCV 检测 "已完成标记"，由运行计数决定

---

## 十一、退出队伍

```
exitTeam():
  repeat(10):
    CHAT_ICON / RECRUIT_TAB? → 已回到大厅 → return
    EXIT_CONFIRM? → click → delay(1000ms)
    BACK_BUTTON? → click → delay(1200ms)
    UNKNOWN? → clickOutside()
```
