## 二、GamePhase 页面模型（v2.2）

> GamePhase 从"节点/动作"重构为"页面"模型，每个 phase 对应 templates/ 下的一个子目录。

### 页面定义

```kotlin
enum class GamePhase {
    IDLE,
    LOBBY,          // templates/lobby/
    CHAT,           // templates/chat/
    RECRUIT_LIST,   // templates/recruit_list/
    TEAM_ROOM,      // templates/team_room/
    FIGHT,          // templates/fight/
    SETTLEMENT,     // templates/settlement/
    RECOVERY,
    DONE
}
```

### 页面迁移

```
IDLE → LOBBY → CHAT → RECRUIT_LIST → TEAM_ROOM → FIGHT → SETTLEMENT → LOBBY...
  ↑                                                            │
  └────────────────────────────────────────────────────────────┘
  当 activeGrades.isEmpty() → DONE
```

### Phase → 方法映射

| Phase | 方法 | 返回目标 |
|-------|------|---------|
| `IDLE, LOBBY, CHAT, RECRUIT_LIST` | `phaseNavigateAndScan()` | `LOBBY / TEAM_ROOM / DONE` |
| `TEAM_ROOM` | `phaseValidate()` | `FIGHT / SETTLEMENT / LOBBY / DONE` |
| `FIGHT` | `phaseBattle()` | `SETTLEMENT / LOBBY / DONE` |
| `SETTLEMENT` | `phaseClaim()` | `IDLE / DONE` |
| `RECOVERY` | delay(1500ms) → | `IDLE` |

## 三、业务流程规范（核心）

### 3.1 核心业务规则

#### 等级-级别映射

| 悬赏等级 | 队伍房间建议等级 | 等级图标模板 |
|---------|----------------|------------|
| D | lv30 | `lv30.png` |
| C | lv40 | `lv40.png` |
| B | lv60 | `lv60.png` |
| A | lv80 | `lv80.png` |
| A+ | lv80 | `lv80.png` |
| S | lv90 | `lv90.png` |
| S+ | lv90 | `lv90.png` |
| SS | lv100 | `lv100.png`（待补充） |
| SS+ | lv125 | `lv125.png` |
| NSS+/NS/NA | lv125 | `lv125.png` |

#### 用户勾选集合规则

用户通过 UI 勾选一组悬赏等级（如 {A, S}），构成 **activeGrades**。全部业务判定基于集合，而非单个级别：

- **招募列表扫描**: 等级图标匹配集合内任意一个 → 判定命中 → 点击加入按钮
- **队伍房间校验**: 房间显示的建议等级（lv80/lv90 等）匹配集合内任意一个 → 判定通过 → 点击准备
- **完成标记**: 某个等级跑完目标次数后，从 activeGrades 中移除
- **停止条件**: activeGrades 为空 → 所有悬赏已完成 → DONE

#### 识别频率

| 模式 | 周期 | 常量 | 适用场景 |
|------|------|------|---------|
| 快速模式 | **100ms** | `FAST_INTERVAL_MS` | 招募列表扫描、战斗检测 |
| 普通模式 | **1000ms** | `NORMAL_INTERVAL_MS` | 导航、等待、结算 |

#### 点击频率

| 动作 | 点击后延迟 | 说明 |
|------|-----------|------|
| 加入/准备/导航点击 | **1000ms** (`POST_CLICK_DELAY`) | 等待 UI 过渡动画完成 |
| 关闭弹窗 | 300~800ms | 弹性延迟，视弹窗类型 |
| 返回按钮 | 800~1200ms | 过渡动画较长 |
| 页签刷新 | 200~300ms | 快速切换 |
| 技能释放 | min **500ms** 冷却 | 防止重复触发 |

#### 完整循环总览

```
runLoop:
  Phase: IDLE/LOBBY/CHAT/RECRUIT_LIST → phaseNavigateAndScan()
    ├─ ensureRecruitView() → 导航到招募列表
    └─ 100ms 全屏检查循环（三块逻辑）
       ├─ ① OUT_OF_RANGE_RECRUIT（超出范围悬赏点击刷新）
       ├─ ② validation（队伍房间校验 + 等级校验 + 准备）
       └─ ③ normal scan（等级图标匹配 + 邀请检测 + TAB刷新）
    return LOBBY / TEAM_ROOM / DONE

  Phase: TEAM_ROOM → phaseValidate()
    └─ 等待 BATTLE_WARNING（15s 超时）
    return FIGHT / SETTLEMENT / LOBBY

  Phase: FIGHT → phaseBattle()
    └─ 检测结算/胜利/失败 + 技能释放
    return SETTLEMENT / LOBBY

  Phase: SETTLEMENT → phaseClaim()
    └─ 关闭结算弹窗 → 领奖 → 回到大厅
    └─ 等级完成 → 从 activeGrades 移除
    return IDLE / DONE

  RECOVERY → delay(1500ms) → IDLE

  异常捕获 → globalFailCount++ → RECOVERY
  globalFailCount >= 3 → 停止 + dump
```

### 3.2 RECRUIT_LIST 阶段 — 详细流程

这是整个流水线的核心阶段。在一个 **100ms 循环**内处理：招募列表刷新 → 等级匹配 → 加入队伍 → 校验级别 → 点击准备。

#### 循环三块逻辑

```
100ms loop {

  ── ① OUT_OF_RANGE_RECRUIT 检测 ─────────────
  条件: currentBounty == null && refreshCooldown <= 0
  检测: matchTemplate(OUT_OF_RANGE_RECRUIT, 阈值 0.5)
  命中: → click(检测坐标) → 跳到最新悬赏列表
       → noMatchCycles = 0
       → refreshCooldown = 10 (1s 冷却防无限刷新)
       → continue

  ── ② Validation 校验模式 ────────────────────
  条件: currentBounty != null（已点击加入，进入队伍房间）

  子步骤（每100ms执行）:
    1. DAILY_LIMIT? → exitTeam() → currentBounty = null → continue
    2. SCOPE_RECRUIT 检测?（还在招募列表?）
       → 命中 → 加入失败异常 → currentBounty = null
       → return GamePhase.LOBBY（扔回主流程重新导航）
    3. READY_BUTTON?
       └─ 命中 → matchAnyLevelIcon(activeGrades) 对全部勾选等级校验
          ├─ 命中任一 → 记录actualGrade → 点击准备 → return TEAM_ROOM
          └─ 无匹配 → 级别不在勾选范围 → exitTeam() → 重置 → continue
    4. 以上皆否 → delay(100ms) → continue

  ── ③ Normal Scan 普通扫描模式 ───────────────
  条件: currentBounty == null

  子步骤:
    1. matchAnyGrade(screen, remaining)
       对 remaining（用户勾选且未完成的等级列表）逐个匹配等级图标
       └─ 命中 → currentBounty = grade
               → matchTemplate(JOIN_BUTTON, 全局匹配)
               ├─ 找到 → click → "X悬赏可见，点击加入" → noMatchCycles=0 → continue
               └─ 未找到 → "未找到加入按钮" → currentBounty=null → continue

    2. 每周期 noMatchCycles++
       达到 3周期无匹配:
         a. READY_BUTTON? → 被邀请检测
            ├─ 命中 + matchAnyLevelIcon(activeGrades) 匹配
            │  → currentBounty = matchedGrade → continue（下周期进校验分支）
            └─ 命中 + 无等级匹配 → exitTeam() → continue
         b. SCOPE_RECRUIT 检测 CHAT_ICON/RECRUIT_TAB?
            → 离开招募界面 → return GamePhase.LOBBY
         c. 仍无变化 → performTabRefresh()（切页签刷新）
}

退出循环返回: DONE
```

#### 关键判定规则

| 场景 | 判定方式 | 动作 |
|------|---------|------|
| 超出范围悬赏出现 | `OUT_OF_RANGE_RECRUIT` 模板匹配 | 点击该按钮刷新列表 |
| 等级图标匹配 | `matchAnyGrade` 遍历 remaining | 设置 currentBounty → 检测 JOIN_BUTTON |
| 点击加入按钮 | `JOIN_BUTTON` 全局模板匹配 | 点击（可能点到其他悬赏的按钮） |
| 加入后在校验模式 | `currentBounty != null` | 每100ms全量检测 |
| 加入失败（仍在招募列表） | `detectForPhase(SCOPE_RECRUIT)` | 清 currentBounty → 扔回主流程 |
| 已达上限弹窗 | `DAILY_LIMIT` 模板匹配 | exitTeam 退出 |
| 准备按钮出现 | `READY_BUTTON` 模板匹配 | 校验等级后点击 |
| 房间等级校验 | `matchAnyLevelIcon(activeGrades)` | 匹配任一勾选等级 → 准备；都不匹配 → 退出 |
| 被邀请进队 | 3周期无匹配 → `READY_BUTTON` 检测 | 等级校验通过 → 进入校验分支 |
| 3周期无任何匹配 | `noMatchCycles >= 3` | TAB 刷新或重新导航 |

### 3.3 招募列表刷新机制

**触发条件:**
1. `OUT_OF_RANGE_RECRUIT` 匹配（点击该按钮直接刷新到最新列表）
2. 连续 3 周期(300ms)无等级匹配 → `performTabRefresh()`（页签切换）

**执行序列（`performTabRefresh`）:**

```
① 检测 CHAT_TAB (templates/chat/private_chat.png)
   匹配 → click → delay(200ms)
   不匹配 → 固定坐标 (0.5f, 0.12f) → delay(200ms)
② 检测 RECRUIT_TAB (templates/chat/team_recruit.png)
   匹配 → click → delay(300ms)
```

**边界约束:**
- `currentBounty != null` 时不触发 OUT_OF_RANGE 检测（已在队伍房间）
- `refreshCooldown` 刷新后 1s 冷却，防止无限刷新循环
- `currentBounty != null` 时不触发 TAB 刷新

---

### 3.4 等级匹配 → 加入队伍

**匹配范围:** `remainingGrades(ctx)` — 用户勾选且未完成次数的等级列表

**匹配方式:** `matchAnyGrade()` — 按 activeGrades 顺序逐个匹配等级图标

**匹配后动作:**

```
① matchAnyGrade 命中 grade → currentBounty = grade
② 全屏检测 JOIN_BUTTON (全局模板匹配 minMaxLoc)
   ├─ 命中 → click → "X悬赏可见，点击加入"
   └─ 未命中 → 跳过该等级 → currentBounty = null
③ continue → 下周期进入 ② 校验模式（currentBounty 已设置）
```

**边界规则:**
- 只匹配 `remainingGrades` 内的等级（勾选且未完成），未勾选的等级不响应
- `JOIN_BUTTON` 是全局匹配，可能点到其他悬赏的按钮 → 校验阶段用 `matchAnyLevelIcon(activeGrades)` 兜底验证
- 日志输出 `"X悬赏可见，点击加入"` 仅表示 X 等级图标可见且点击了加入按钮，**不代表进入了 X 的队伍**

---

### 3.5 队伍房间 → 准备

**进入方式:**
1. **主动加入**: 匹配等级 → 点击加入按钮 → `currentBounty != null` → 校验分支
2. **被动邀请**: 3周期无匹配 → 检测到 `READY_BUTTON` + 等级校验通过 → `currentBounty` 设值

**准备动作序列:**

```
① READY_BUTTON 检测到
② matchAnyLevelIcon(screen, activeGrades)
   ├─ 命中某个等级 → log "等级匹配 X (lvX)，点击准备"
   │  → click(READY_BUTTON)
   │  → delay(1000ms)
   │  → return GamePhase.TEAM_ROOM（进入等待战斗）
   └─ 无匹配 → "队伍级别不在勾选范围内，退出"
      → exitTeam()
      → currentBounty = null
      → continue（回到普通扫描）
```

**异常处理（准备前）:**

| 检测 | 动作 |
|------|------|
| `DAILY_LIMIT` 弹窗 | `exitTeam()` → 退出队伍 → 重置 currentBounty |
| `SCOPE_RECRUIT` 命中 | 加入失败异常 → 返回主流程重新导航 |
| 以上皆否 | 保持 100ms 循环等待 |

**等级校验业务规则:**
- 校验用 `matchAnyLevelIcon(screen, ctx.activeGrades)` 而非 `currentBounty`
- 例如用户勾选 {A, S}，房间显示 lv80 或 lv90 都算匹配成功
- 不匹配 → 退出队伍 → 回到普通扫描继续找其他悬赏

---

### 3.6 TEAM_ROOM 阶段（等待战斗）

由 `phaseValidate()` 处理，在 `phaseNavigateAndScan` 点击准备后进入。

**等待 BATTLE_WARNING（15s 超时）:**

```
while (< 15s):
  BATTLE_WARNING? → click → delay(1000ms) → return FIGHT
  WAITING_SCREEN? → continue（正常倒计时）
  SETTLEMENT_POPUP / CONFIRM_BUTTON? → 战斗已结束 → SETTLEMENT
  CHAT_ICON / RECRUIT_TAB? → 回到大厅 → IDLE
  每 5s → log "等待中... (Xs)"
15s 超时 → exitTeam() → IDLE
```

- 检测周期: **100ms**
- 超时: **15s** (`WAIT_BATTLE_TIMEOUT_MS`)

---

### 3.7 FIGHT 阶段（战斗）

```
while (true):
  SETTLEMENT_POPUP? → return SETTLEMENT
  CONFIRM_BUTTON? → return SETTLEMENT
  DEFEAT_POPUP? → return SETTLEMENT
  每 500ms:
    ULTIMATE_SKILL (阈值 0.6) → click + 200ms 冷却
    WEAPON_SKILL (阈值 0.6) → click + 200ms 冷却
  10次(1s)无匹配 → SCOPE_BATTLE 判定
    CHAT_ICON/RECRUIT_TAB? → 大厅异常 → IDLE
```

- 检测周期: **100ms**
- 技能冷却: **500ms**

---

### 3.8 SETTLEMENT 阶段（结算领奖）

```
while (< 300轮):
  SETTLEMENT_POPUP(黑色遮罩)? → clickOutside() → delay(800ms)
  CONFIRM_BUTTON(确定)? → click → delay(1000ms)
  CHAT_ICON? → 回到大厅 → break
  3次无匹配 → SCOPE_CLAIM 判定
```

**完成标记:**
```
回到大厅后:
  runCounts[grade] += 1
  log "X 完成 N/M"
  if (runCounts >= targetRuns):
    activeGrades = activeGrades - grade  ← 从集合删除
    log "X 全部完成，从集合移除"
  currentBounty = null
  if (activeGrades.isEmpty()):
    log "所有悬赏已完成！"
    return DONE
  return IDLE
```

**关键规则:**
- 等级完成后从 activeGrades 移除，后续扫描不再包含该等级
- 所有等级移除完毕 → `activeGrades.isEmpty()` → DONE
- 不需要用 OpenCV 检测 "已完成标记"，完成状态由运行计数决定

---

### 3.9 退出队伍流程

```
exitTeam():
  repeat(10):
    SCOPE_EXIT 检测:
      CHAT_ICON / RECRUIT_TAB? → 已回到大厅 → return
      EXIT_CONFIRM? → click → delay(1000ms)
      BACK_BUTTON? → click → delay(1200ms)
      UNKNOWN? → clickOutside() → delay(800ms)
```

## 五、检测模式

### 两种检测速率

| 模式 | 周期 | 适用场景 |
|------|------|---------|
| 普通模式 | 1000ms | 导航、等待战斗、结算阶段 |
| 快速模式 | 100ms | 招募列表扫描（核心）、战斗检测 |

### 线性链检测 vs 全量扫描

| 特性 | 线性链检测 | 全量扫描 |
|------|-----------|---------|
| 触发时机 | 默认 | 兜底 / 异常恢复 |
| 检测范围 | 传入的 ScreenState 列表 | `detectionOrder` (所有状态) |
| 停止条件 | 第一个匹配 | 第一个匹配或全不匹配 |
| 方法 | `detectForPhase(screen, scope)` | `detectWithCoord(screen)` |

### 3-Miss 兜底机制

连续 N 次线性检测未匹配 → 触发一次范围判定:
- 扫描阶段: 3 周期无等级匹配 → TAB 刷新（非断言，纯刷新）
- 结算阶段: 3 次 CONFIRM_BUTTON 未匹配 → SCOPE_CLAIM 判定
- 战斗阶段: 10 次无匹配 → SCOPE_BATTLE 判定

### 3-Strike 全局兜底

`globalFailCount` 累计异常 → 达到 3 次 → 脚本停止 + 崩溃日志写文件

## 七、服务层设计

### GameManager

```kotlin
object GameManager {
    state: StateFlow<ScriptState>  // IDLE / RUNNING / PAUSED
    logEvents: SharedFlow<String>  // 日志流 → UI

    fun toggleScript(context)  // start / pause / resume
    fun startScript(context)   // 等待 MediaProjection → 启动 WorkflowEngine
    fun pauseScript()          // cancel job → PAUSED
    fun resumeScript(context)  // 恢复 MediaProjection → 启动新 WorkflowEngine
    fun stopScript()           // cancel job → release MediaProjection → IDLE
    fun updateBountyConfigs(configs)
    fun getSelectedGrades(): List<BountyGrade>
}
```

### GameContext

```kotlin
data class GameContext(
    currentPhase: GamePhase,
    activeGrades: List<BountyGrade>,          // 用户勾选的等级
    runCounts: MutableMap<BountyGrade, Int>,  // 各等级已跑次数
    targetRuns: Map<BountyGrade, Int>,        // 用户设定目标次数（来自 config）
```

### 异常处理链

```
WorkflowEngine.runLoop()
  └─ try { phaseXxx(ctx) } catch (非 CancellationException)
       └─ globalFailCount++
       └─ ctx.currentPhase = RECOVERY
       └─ RECOVERY → delay(1500ms) → IDLE
       └─ globalFailCount >= 3 → writeCrashLog() → 停止
```

## 八、已知问题 & 已修复

### 已修复（v2.3）

1. ~~**校验分支 10-cycle 节拍**~~ — ✅ 移除 `noMatchCycles % 10 != 0` 跳过逻辑，校验模式每 100ms 全量检测，不再 1s 一次。

2. ~~**加入失败死循环**~~ — ✅ 校验分支新增 `SCOPE_RECRUIT` 检测：加入按钮点击后若屏幕仍在招募列表，判定为异常并返回主流程重新导航。不使用超时计数器，直接检测屏幕状态。

3. ~~**TEAM_FULL/TEAM_COMPLETED 误检测**~~ — ✅ 移除不可靠的 OpenCV 模板匹配。完成了的判定改由结算阶段的运行计数决定。

4. ~~**等级校验只针对 currentBounty**~~ — ✅ 改为 `matchAnyLevelIcon(screen, ctx.activeGrades)` 对全部用户勾选等级匹配。勾选 {A,S} 时，房间显示 lv80 或 lv90 都算匹配。

5. ~~**完成标记逻辑**~~ — ✅ 结算完成后从 `activeGrades` 移除该等级，不再参与后续扫描。全部移除后 `activeGrades.isEmpty()` → DONE。

6. ~~**RECRUIT_EXCEPTION → OUT_OF_RANGE_RECRUIT**~~ — ✅ 重命名，检测到后直接点击该按钮跳到最新列表（替换 TAB 切页刷新）。

7. ~~**异常检测跳过校验模式**~~ — ✅ 加上 `ctx.currentBounty == null` 条件，校验模式中不检测超出范围悬赏，消除日志干扰。

8. ~~**"X匹配，点击加入"日志误导**~~ — ✅ 改为 "X悬赏可见，点击加入"，不再将图标识别和按钮归属绑在一起。

9. ~~**DEBUG级相似度日志刷屏**~~ — ✅ 删除匹配失败的 DEBUG 日志，只保留匹配成功的 INFO 日志。

### 待修复

10. **`globalFailCount` 永不归零** — 是累计计数而非连续失败计数，3 次分散的瞬态异常也会停止脚本。

11. **活动悬赏等级图标模板缺失** — NSS+.png / NS.png / NA.png，影响活动悬赏扫描。

12. **`phaseClaim` 无超时上限** — 300 轮 × 1s = 5 分钟才退出，应加更短的超时机制。

13. **`phaseJoin` 死代码** — JOINING 阶段已不再使用，但 `phaseJoin` 方法仍保留为 dead code。
