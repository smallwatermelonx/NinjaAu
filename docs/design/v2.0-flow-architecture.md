## 一、整体架构

```

│  GamePhase:                                       │
```

## 二、GamePhase 状态机

### 状态定义

```kotlin
enum class GamePhase {
    IDLE,       // 开始 / 回到大厅等待下一轮
    SCANNING,   // 【核心】导航 + 扫描 + 加入 + 准备（单循环处理全部）
    JOINING,    // 已废弃（保留为死代码）
    VALIDATING, // 准备后等待战斗 + BATTLE_WARNING 检测
    WAITING,    // 保留（映射到 VALIDATING）
    BATTLE,     // 战斗中 + 技能释放
    SETTLEMENT, // 结算领奖
    RECOVERY,   // 异常恢复 (1.5s → IDLE)
    DONE        // 全部完成
}
```

### 状态迁移

```
IDLE ──→ SCANNING ──→ VALIDATING ──→ BATTLE ──→ SETTLEMENT ──→ IDLE
  ↑                                                              │
  └──────────────────────────────────────────────────────────────┘
  当 allCompleted=true → DONE
```

**关键变化:** JOINING 阶段已删除。SCANNING 在一个循环内处理 等级匹配 → 加入队伍 → 点击准备 全流程。

**异常路径:**
- 任一 phase 抛出异常 → catch → `globalFailCount++` → `RECOVERY(1.5s)` → `IDLE`
- `globalFailCount >= MAX_GLOBAL_FAIL(3)` → 停止脚本 + `writeCrashLog()`

### Phase → 方法映射

| Phase | 方法 | 返回目标 |
|-------|------|---------|
| `IDLE, SCANNING` | `phaseNavigateAndScan()` | `SCANNING / VALIDATING / DONE` |
| `VALIDATING, WAITING` | `phaseValidate()` | `BATTLE / SETTLEMENT / IDLE / SCANNING / DONE` |
| `BATTLE` | `phaseBattle()` | `SETTLEMENT / IDLE / DONE` |
| `SETTLEMENT` | `phaseClaim()` | `IDLE / DONE` |
| `RECOVERY` | delay(1500ms) → | `IDLE` |

## 三、业务流程规范（核心）

### 3.1 全部循环总览

```
┌─────────────────────────────────────────────────────────────┐
│                      runLoop (大循环)                         │
│  while not DONE and globalFailCount < 3:                     │
│    phaseNavigateAndScan() → 导航 + 扫描 + 加入 + 准备          │
│    phaseValidate()         → 等待BATTLE_WARNING → 点击进战斗  │
│    phaseBattle()           → 战斗 + 技能释放                  │
│    phaseClaim()            → 结算领奖                         │
│    回到 IDLE / DONE                                          │
│    异常 → RECOVERY → IDLE                                    │
└─────────────────────────────────────────────────────────────┘
```

### 3.2 SCANNING 阶段（核心单循环）

这是整个流水线中最复杂的阶段。在一个 100ms 循环内处理：招募列表异常处理 → 等级匹配 → 加入队伍 → 进入房间 → 点击准备。

#### 循环优先级

```
while (100ms / 周期) {

  ── STEP 1: 异常检测 ──────────────────────────
  检测 RECRUIT_EXCEPTION (exception.png, 阈值 0.7)
  命中且 currentBounty == null → TAB刷新（见 §3.3）→ continue
  命中但 currentBounty != null → 忽略（已在加入流程）

  ── STEP 2: 加入/队伍流程 ─────────────────────
  if (currentBounty != null) {
    JOIN_BUTTON 可见? → 点击 → continue
    JOIN_BUTTON 消失? →
      READY_BUTTON 可见? → 点击 → 返回 VALIDATING
      DAILY_LIMIT/TEAM_COMPLETED/TEAM_FULL? → exitTeam → 重置
      过渡中 → delay(1000ms) → continue
  }

  ── STEP 3: 普通扫描 ─────────────────────────
  READY_BUTTON 可见? → 点击（被邀请直达）→ 返回 VALIDATING
  SCOPE_RECRUIT 检查 → 离开/UNKNOWN 处理
  等级匹配（matchAnyGrade）→ 匹配到? → 见 §3.4
  3 周期无匹配 → TAB刷新（最多 5 轮）
}
```

#### 关键实现细节

| 方面 | 行为 |
|------|------|
| **截图频率** | 每 100ms 一次（`FAST_INTERVAL_MS`） |
| **点击后** | 不退出循环，`continue` 继续下一轮 100ms 检测 |
| **一旦加入** | `currentBounty` 置为匹配到的等级，进入 STEP 2 |
| **JOIN_BUTTON 反复点击** | 只要 JOIN_BUTTON 还在画面中就反复点击（高失败率） |
| **JOIN_BUTTON 消失** | 判定已进入队伍房间，切换至 READY_BUTTON 检测 |
| **进入房间后** | 1s 慢速检测 READY_BUTTON（不再 100ms 空转） |
| **异常弹窗** | DAILY_LIMIT/TEAM_COMPLETED/TEAM_FULL → exitTeam → 重置 |
| **无匹配超时** | 3 周期(300ms)无等级匹配 → TAB 刷新 |
| **刷新上限** | 5 次刷新仍无匹配 → 回到导航 |

---

### 3.3 TAB 刷新机制

**触发条件:**
1. `RECRUIT_EXCEPTION` 匹配（最高优先级）
2. `SCOPE_RECRUIT` 返回 UNKNOWN（无法识别的招募界面）
3. 连续 3 周期(300ms)无等级匹配

**执行序列（`performTabRefresh`）:**

```
① 检测 CHAT_TAB (templates/chat/private_chat.png)
   匹配 → click → delay(200ms)
   不匹配 → 固定坐标 (0.5, 0.12) → delay(200ms)
② 检测 RECRUIT_TAB (templates/chat/team_recruit.png)
   匹配 → click → delay(300ms)
```

**边界约束:**
- 最多连续刷新 5 轮，之后回到导航阶段
- `currentBounty != null` 时不触发刷新（已在加入流程）

---

### 3.4 等级匹配 → 加入流程

**匹配范围:** `remainingGrades(ctx)` — 用户勾选且未完成次数的等级列表

**匹配方式:** `matchAnyGrade()` — 按优先级顺序匹配等级图标

**匹配后动作序列:**

```
① matchAnyGrade 命中 → currentBounty = grade
② 全屏检测 JOIN_BUTTON → 匹配到则 click
③ JOIN_BUTTON 不匹配 → 直接 click 等级图标位置（fallback）
④ continue → 保持 100ms 循环
```

**边界约束:**
- 只匹配用户勾选的等级（未勾选的等级即使出现在画面上也不响应）
- 匹配成功后不退出循环，继续以 100ms 检测
- 后续循环负责：继续点 JOIN_BUTTON → 等待消失 → 检测 READY_BUTTON

---

### 3.5 队伍房间 → 准备

**进入方式:**
1. **正常流程**: 匹配等级 → 点击加入 → JOIN_BUTTON 消失 → READY_BUTTON 出现
2. **被动进入**: 被邀请进队 → 循环中直接检测到 READY_BUTTON

**准备动作:**

```
READY_BUTTON 匹配 (templates/team_room/prepare.png) → click
等级校验: matchLevelIcon(screen, grade) → 仅日志，不阻断
delay(POST_CLICK_DELAY = 1000ms)
return GamePhase.VALIDATING
```

**异常处理（准备前）:**
- `DAILY_LIMIT` → `exitTeam()` → 重置 `currentBounty`
- `TEAM_COMPLETED` → `exitTeam()` → 重置
- `TEAM_FULL` → `exitTeam()` → 重置

---

### 3.6 VALIDATING 阶段（等待战斗）

**阶段1: READY_BUTTON 检测（冗余兜底）**
- 周期: 1s
- 上限: 15 次
- 点击准备后进入阶段2

**阶段2: 等待 BATTLE_WARNING**
- 周期: 100ms
- 超时: 15s

```
BATTLE_WARNING (warning.png, 阈值 0.7)?
  → click → delay(1000ms) → 返回 BATTLE

WAITING_SCREEN? → continue（正常倒计时）
SETTLEMENT_POPUP / CONFIRM_BUTTON? → 战斗已结束 → SETTLEMENT
CHAT_ICON / RECRUIT_TAB? → 回到大厅 → IDLE
15s 超时 → exitTeam() → IDLE
```

---

### 3.7 BATTLE 阶段（战斗）

- 周期: 100ms
- 冷却: 500ms（技能释放间隔）

```
SETTLEMENT_POPUP → SETTLEMENT
CONFIRM_BUTTON → SETTLEMENT
DEFEAT_POPUP → SETTLEMENT
每 500ms:
  ULTIMATE_SKILL (r_ziyuan.png, 阈值 0.6) → click
  WEAPON_SKILL (wopen_shedao.png, 阈值 0.6) → click
10 次无匹配 → SCOPE_BATTLE 判定 → IDLE（异常回到大厅）
```

---

### 3.8 SETTLEMENT 阶段（结算领奖）

- 周期: 1s
- 兜底: 3 次无匹配 → SCOPE_CLAIM

```
clickOutside() → delay(300ms) → 关闭弹窗/遮罩
检测 CONFIRM_BUTTON → click
检测 CHAT_ICON → 回到大厅 → break
回到大厅 → 计数 +1
allCompleted? → DONE : IDLE
```

## 五、检测模式

### 两种检测速率

| 模式 | 周期 | 适用场景 |
|------|------|---------|
| 普通模式 | 1000ms | 导航、等待战斗、结算阶段 |
| 快速模式 | 100ms | 招募列表扫描（核心）、战斗检测 |

### 线性链检测 vs 全量扫描

| 特性 | 线性链检测 | 全量扫描 |
|------|-----------|---------|
| 触发时机 | 默认 | 兜底 / 异常恢复 |
| 检测范围 | 传入的 ScreenState 列表 | `detectionOrder` (所有状态) |
| 停止条件 | 第一个匹配 | 第一个匹配或全不匹配 |
| 方法 | `detectForPhase(screen, scope)` | `detectWithCoord(screen)` |

### 3-Miss 兜底机制

连续 N 次线性检测未匹配 → 触发一次范围判定:
- 扫描阶段: 3 周期无等级匹配 → TAB 刷新（非断言，纯刷新）
- 结算阶段: 3 次 CONFIRM_BUTTON 未匹配 → SCOPE_CLAIM 判定
- 战斗阶段: 10 次无匹配 → SCOPE_BATTLE 判定

### 3-Strike 全局兜底

`globalFailCount` 累计异常 → 达到 3 次 → 脚本停止 + 崩溃日志写文件


## 八、已知问题

### 优先级高

1. **`BountyConfig.targetRuns` 被忽略** — `buildContext()` / `remainingGrades()` / `allCompleted` 均使用 `grade.defaultRuns` 而非 `config.targetRuns`，用户自定义次数不生效。

2. **导航缺少 DAILY_LIMIT / DEFEAT_POPUP 兜底** — `SCOPE_NAVIGATE` 中没有这两个状态，启动时若处于对应弹窗会一直 UNKNOWN。

3. **结算阶段 clickOutside 顺序** — `phaseClaim` 先 `clickOutside()` 再对旧截图检测 CONFIRM_BUTTON，若 clickOutside 点掉弹窗则坐标无效。

### 优先级中

4. **`globalFailCount` 永不归零** — 是累计计数而非连续失败计数，3 次分散的瞬态异常也会停止脚本。

5. **`clickOutside()` 硬编码 540x1200** — 仅适配 ~1080p 分辨率，不同屏幕比例会点偏。

6. **活动悬赏等级图标模板缺失** — NSS+.png / NS.png / NA.png，影响活动悬赏扫描。

### 优先级低

7. **`phaseClaim` 无超时上限** — 300 轮 × 1s = 5 分钟才退出，应加更短的超时机制。

8. **SCANNING 循环分支过多** — 单方法内包含 异常→加入→队伍→扫描 多条分支，可读性差，后续应拆分子方法。

9. **`phaseJoin` 死代码** — JOINING 阶段已不再使用，但 `phaseJoin` 方法仍保留为 dead code。
